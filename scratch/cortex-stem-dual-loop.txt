## Workflow

Make progressive planning:  

1. Analyze L0 (User Input) & Context: Deconstruct the request. Explore the codebase for existing patterns, identify architectural trade-offs; and research external sources, ask clarifying questions if necessary.
2. Draft L1 (High-Level Strategy): Define the technical approach. Establish the architectural design, key technical decisions, and dependency requirements. It's suggested to interact with user actively at this stage.
3. Draft L2 (Low-level Design): Iterate on L1 until approved. Then, define the low-level specifics: interfaces, data structures, and algorithms.
4. Draft L3 (Implementation Plan): Iterate on L2 until approved. Then, outline the execution roadmap: pseudo-code, step-by-step logic, boundaries, and test plans.
5. Execute Implementation: Iterate on L3 until approved. Implement the solution strictly following the comprehensive L3 plan.
6. Finalize Documentation: Compile the output and write `docs/task/RESULT.md`.  

Notes:

- Write each stage of plan to file `docs/task/<task-name>/L<X>-PLAN.md`. Spilt into multiple files if needed.
- MUST ask user for explicit approval before getting into next planning.
- Make use of sub agent to reduct your cognitive load.

## Task
Introduce Cortex Loop Architecture.

1. Pathway Ownership Refactor
    - Afferent Pathway: 
        - Transfer ownership of the afferent receive handle from Stem to Cortex.
        - Cortex becomes the sole consumer of inbound senses.
    - Efferent Pathway:
        - Introduce an EfferentPathway module responsible for broadcasting acts.
        - Continuity, Spine, Stem and main retain their respective receive handles.
        - A consumer of the efferent pathway must not block other consumers.
        - Remove Act dispatching logic from Stem.

2. Cortex Control Flow Refactor
    - The Cortex now owns the loop (driven by tick / sense), not Stem.
    - The Cortex Primary is implemented on top of the AI Gateway chat thread (streaming response).
    - A Cortex Primary turn is driven by proprioception updates, ticks, and incoming senses.
    - Act Emssion Model:
        - The Cortex primary emits acts via structured tool-calling (replace prompt-based dispatching).
        - Remove the act descriptor catalog from the system prompt.
        - Each act tool invocation contains `payload`, `wait_for_sense: bool`;
        `wait_for_sense` determines whether the primary should block until the corresponding sense arrives.
        - Act Descriptor may optionally declare emitted senses (array of sense ids).
        - Extend Sense with an optional `act_instance_id`.
    - Goal Forest Patching:
        - Add parameter `reset: bool`. If reset is true:
            1. Remove all messages from the first user message up to the current tool call.
            2. Replace the goal forest section in the system prompt.
            3. Resend the updated system message.
        - Numbering in tool parameters is optional, as it will be generated automatically (though automatic generation is not yet implemented).
    - Sense:
        - Remove automatic sense transformation (the Sense input helper). Incoming senses must be formatted as:
        ```
        [internal-sense-instance-id]. [sense-id] `compacted JSON payload (truncated if it exceeds the configured byte limit)`\n
        ...
        ```
        - Merge expand-senses-with-sub-agent and expand-senses-raw into a single tool: expand-senses, parameterized by `mode: raw | sub-agent`.

3. Cortex State Refactor
    - Physical State:
        - Remains owned and maintained by Stem.
        - Expose as `Arc<RwLock<PhysicalState>>` to ensure thread-safe shared access.
        - Proprioception remains in the system prompt; the <proprioception> section must be refreshed before dispatching any new senses to the Primary.
    - Cognition State:
        - Persistence is achieved by Cortex calling Continuity directly.
        - Remove cognition state relevant logic from Stem.
        - Remove L1 Memory (focal awareness) since its responsibilities are absorbed by the persistent thread and the goal forest.
    - Goal Forest (part of cognition state):
        - Convert goal forest helper into a global singleton.
        - Implement deterministic `to_input_ir`, remove LLM-based transformation.
