# L3 Implementation Stages For This Task

All stages below are implementation stages of this single Moira task. They are not separate product phases.

## Stage 0. Task Buffer And Scaffold

### Objective

Finish the docs-first workflow and lock the implementation packet before code changes begin.

### Likely Paths

- `docs/task/moira-v1/*`

### Done When

- `L0`, `L1`, `L2`, `L3`, and `LACHESIS` are coherent enough to guide implementation
- assumptions, defaults, and unresolved questions are explicit
- no authoritative docs are half-updated

## Stage 1. Lachesis Minimal Ingest

### Objective

Start code implementation with the minimum Moira scaffold required to receive OTLP logs from Core, store them, and support run- and tick-scoped inspection.

### Scope

- `moira/*` scaffold
- OTLP gRPC logs receiver
- DuckDB `raw_events` table
- DuckDB `runs` projection
- DuckDB `ticks` projection
- minimal query API
- Loom run list
- Loom tick timeline
- Loom tick detail with Cortex / Stem / Spine tabs
- raw-event inspector for the selected tick

### Done When

- a running Core can stream logs into Moira
- operators can inspect one run and one selected tick from Moira without external collector/UI tooling
- selected tick detail is reconstructed from raw events plus `runs`/`ticks`, not from a larger precommitted projection lattice

### Stage Gate

- confirm which missing Core fields actually block run list, tick timeline, or tick detail before expanding the Core contract further
- do not introduce dedicated Stem/Spine projection tables unless the first slice proves raw-first querying is not enough
- do not ask release-artifact questions here

## Stage 2. Core Structured Log Upgrade

### Objective

Close the Core structured-log gaps exposed by the first Loom run/tick inspection slice.

### Scope

- add or tighten canonical event families only where Stage 1 surfaces prove the need
- preserve enough raw payload for debugging
- avoid new broad MVP test work unless a contract change would otherwise be unreadable or unsafe

### Done When

- run list, tick timeline, and tick detail do not depend on fragile free-form parsing as their primary contract
- the remaining dedicated Stem/Spine/goal-forest surfaces have the minimum structured fields they need to land cleanly later

### Stage Gate

- decide which tick-boundary records Lachesis materializes as first-class tables
- decide which large payloads stay raw-only versus summarized

## Stage 3. Atropos Minimal Supervision

### Objective

Wake and stop a selected local Core build under Moira control.

### Scope

- launch command construction
- receiver-ready gating
- PID/run-state tracking
- graceful stop and explicit force-kill
- app-exit behavior that stops supervised Core

### Done When

- Moira can start a local Core build, observe it, and stop it deterministically

### Stage Gate

- no new release-manifest questions until supervision works with a known local build

## Stage 4. Loom Observability Expansion

### Objective

Expand from the first run/tick slice into dedicated Beluna-native inspection workflows only where the initial slice proves the need.

### Scope

- Goal-forest compare
- dedicated Stem signal timeline
- dedicated Spine topology and dispatch view
- any additional derived tables justified by those views

### Done When

- the requested Cortex/Stem/Spine primitives are visible without leaving Moira
- new derived tables exist only where they clearly pay for themselves in Loom simplicity or query cost

### Stage Gate

- confirm which views actually need additional derived tables instead of assuming them

## Stage 5. Clotho Artifact Management

### Objective

Move from “run whatever binary is already on disk” to versioned artifact management.

### Scope

- local source-folder compile workflow
- GitHub release discovery
- checksum verification
- version-isolated install directories
- selected artifact binding in Atropos wake path

### Done When

- Moira can compile from a local Core source folder for development
- Moira can install, verify, select, and run a specific Core version

### Stage Gate

- release asset naming, checksum file naming, and trust contract are decided immediately before this stage

## Stage 6. Config And Promotion Hardening

### Objective

Make the control plane usable for repeated iteration and promote stable truths upward.

### Scope

- JSONC profile management
- Core schema validation
- authoritative doc updates
- monitor retirement or redirection strategy

### Done When

- the same workflow can be repeated across multiple Core versions and profiles
- stable boundaries are promoted into `20-product-tdd`, `30-unit-tdd`, and `40-deployment`

## Stage Rule

Each stage should be landable on its own. Avoid stages that simultaneously introduce:

- a new durable boundary
- a new runtime surface
- a new storage contract
- and a new UI abstraction

That combination would hide risk instead of isolating it.

During MVP, do not make broad automated test additions a stage gate by default. Prefer live end-to-end checks until the read models and operator surfaces stop churning.
