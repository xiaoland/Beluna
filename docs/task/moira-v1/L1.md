# L1 Scope Split And Workstreams

## V1 Objective

Land a local binary Moira that can supervise one Beluna Core runtime and provide Beluna-native log observability views that are materially better than raw NDJSON tails or generic backend UIs.

## Task Workflow Alignment

- `L0`: stabilize context, assumptions, and governing anchors in docs.
- `L1`: agree the workstreams and stage order.
- `L2`: decide the target seams, data model, and observability contract.
- `L3`: execute the implementation stages of this same task.

All `Stage 0..N` items below and in `L3.md` are in-scope implementation stages for task completion. They are sequencing devices, not product-scope reductions.

## Explicit Non-Goals For This Packet

- cross-platform parity on day one
- full metrics and traces productization
- unified launcher for every body endpoint app
- durable contract promotion before code proves the design

## Workstreams

### W0. Governance And Cutover

- Define the cutover from `monitor` to `moira`.
- Track which authoritative docs must change once code lands.
- Keep transitional assumptions quarantined in this packet.

### W1. Core Telemetry Contract

- Identify which current logs are too string-heavy for Lachesis.
- Define Beluna-native structured OTLP log families for Cortex, Stem, and Spine.
- Add the minimal event vocabulary required for tick-level and pathway-level inspection.
- Converge product-facing naming from `cycle_id` to `tick` at the observability contract boundary.

### W2. Lachesis Backend

- OTLP logs receiver.
- log normalization and persistence into DuckDB.
- query surface for live and historical views.
- retention and migration design that preserves inspectability.

### W3. Atropos Runtime Supervision

- Core process wake/stop/force-kill.
- run-state tracking and failure reporting.
- receiver-readiness gating so observability comes up before supervised Core launch.

### W4. Clotho Artifact And Config Management

- GitHub release discovery.
- checksum verification.
- version-isolated installs.
- config profile preparation and schema validation.
- local source-folder compilation for development builds.

### W5. Loom UI

- control-plane pages for install/config/run state.
- observability pages for Overview, Cortex, Stem, and Spine.
- live tail plus indexed drilldown workflows.

### W6. Packaging And Promotion

- macOS app packaging shape.
- app-local directory layout.
- promotion of stable conclusions into `10/20/30/40`.

## Dependency Map

- `W1 -> W2 -> W5`
  - rich UI depends on structured Core logs and queryable storage.
- `W2 -> W1`
  - the first Moira slice should expose which additional Core fields are actually missing before expanding the Core log catalog further.
- `W2 -> W3`
  - supervised launches should not start before the ingest path is ready.
- `W4 -> W3`
  - supervised wake depends on a selected Core artifact and config profile.
- `W0` and `W6` cross-cut all streams.

## Recommended Stage Order

Observability is the primary risk reducer, so implementation should not start with artifact plumbing just because Clotho appears first in the mythology.

1. Stage 0: `W0` governance packet and Moira scaffold
2. Stage 1: `W2` first Lachesis vertical slice: raw ingest + `runs`/`ticks` projections + Loom run/tick inspection
3. Stage 2: `W1` close the Core structured-log gaps exposed by the Stage 1 Loom surfaces
4. Stage 3: `W3` Atropos minimal wake/stop over a known local Core build
5. Stage 4: `W2` + `W5` expand into dedicated goal-forest compare, Stem, and Spine views only where raw-first querying proves insufficient
6. Stage 5: `W4` Clotho local-dev compile path, then artifact download/checksum/version isolation
7. Stage 6: `W6` promotion, packaging hardening, monitor retirement

## Why This Order

- It validates the most important product bet first: whether Moira can make Beluna observable in a Beluna-native way.
- It avoids a common failure mode where distribution machinery lands before the runtime is actually inspectable.
- It lets Stage 1 expose which Core fields truly matter in Loom before Stage 2 hardens more event families.
- It lets Atropos and Clotho reuse the same runtime/status surfaces later instead of inventing separate ad hoc state models.
- It keeps late-stage questions late. Release-asset naming and checksum publishing are not allowed to block the early observability stages.
