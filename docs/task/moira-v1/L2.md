# L2 Architecture Seams And Candidate Contracts

## Boundary Statement

- `core` remains authoritative for:
  - typed config schema
  - runtime behavior
  - endpoint routing and outcomes
  - observability emission
- `moira` becomes authoritative for:
  - local Core artifact management
  - local Core supervision
  - OTLP log ingestion and storage
  - human-facing control and inspection UX
- `apple-universal` remains a body endpoint app, not a control-plane owner.

  - top-level module composition
  - command exposure
  - app-wide event wiring

  - checksum verification
  - install isolation
  - local source-folder validation
  - Core build invocation for development runs
  - build output discovery
  - JSONC profile storage
  - Core schema validation

- `lachesis`
  - OTLP gRPC logs receiver

  - DuckDB connection, migrations, retention, query views
  - live subscription
  - indexed lookups

## Candidate Local State Layout

This is a working shape only, not yet a durable contract.

- `installs/<core-version>/`
- `profiles/<profile-name>.jsonc`
- `runtime/current-run.json`
- `telemetry/moira.duckdb`
- `cache/releases/`

The next Lachesis increment should still keep `raw_events` authoritative, but it may add a narrow tick-browse projection if that clearly pays for the human-friendly lane view:

- `raw_events`
- `runs`
- `ticks`
- `tick_lanes` or an equivalent derived browse model if raw-only assembly proves too awkward

## Candidate Data Flow

1. Loom asks Clotho for available artifacts and profiles.
2. Atropos wakes Core using a selected artifact + profile.
3. The selected profile points Core OTLP logs at Lachesis.
4. Lachesis ingests OTLP logs, normalizes Beluna-native fields, and writes raw + indexed rows into DuckDB.
5. Loom queries wake/tick/tick-lane read models derived from DuckDB and subscribes to live updates for active wakes.
6. Metrics and traces remain external handoff surfaces; Loom shows exporter state and links rather than collecting them locally.

## Candidate Structured Event Families Required From Core

These logical families define the next implementation target. They replace the thin summary-first contract with owner-centric families that match the real runtime boundaries. Wire encoding may temporarily use family suffixes such as `request/response` or `bind/outcome`, but the logical contract stays coarse-grained.

### AI-Gateway

- `ai-gateway.request`
  - backend request lifecycle
  - retry and failure detail
  - full request/response payloads where available
  - terminal-state provider request payload retention when adapters cannot expose intermediate bodies cleanly
- `ai-gateway.turn`
  - committed turn boundary
  - full turn messages array
  - usage, finish reason, backend metadata
- `ai-gateway.thread`
  - authoritative thread snapshot
  - thread open or clone/reset visibility
  - post-turn persisted thread state

### Cortex

- `cortex.tick`
  - skipped, gated, admitted, completed, or failed tick status
  - full admitted senses and physical snapshot when a cognition cycle runs
- `cortex.organ`
  - `organ_id`
  - full input/output payloads
  - route, request id, status, and error detail
- `cortex.goal-forest`
  - per-tick snapshot
  - patch or persist result payload
  - context-reset details when applicable

### Stem

- `stem.tick`
- `stem.signal`
  - direction plus transition kind
  - full sense payload when relevant
  - afferent and efferent movement under one stable family
- `stem.dispatch`
  - queue admission
  - dispatch
  - terminal result
  - full act payload
- `stem.proprioception`
  - patch/drop changes owned by the physical-state store
- `stem.descriptor.catalog`
  - snapshot/update/drop at catalog scope
- `stem.afferent.rule`
  - add/remove lifecycle for deferral rules

### Spine

- `spine.adapter`
  - lifecycle state
  - startup and fault visibility
- `spine.endpoint`
  - connect/drop lifecycle
  - channel/session visibility
- `spine.dispatch`
  - binding
  - terminal outcome

## Decision Surfaces Still Open

- whether any adapter can cheaply expose provider-request payloads on every `ai-gateway.request` kind rather than only at terminal states
- whether any Stage 2 Loom view needs a secondary grouping inside one primary lane beyond the currently fixed lane matrix
