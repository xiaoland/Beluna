# L2 Architecture Seams And Candidate Contracts

## Boundary Statement

- `core` remains authoritative for:
  - typed config schema
  - runtime behavior
  - endpoint routing and outcomes
  - observability emission
- `moira` becomes authoritative for:
  - local Core artifact management
  - local Core supervision
  - OTLP log ingestion and storage
  - human-facing control and inspection UX
- `apple-universal` remains a body endpoint app, not a control-plane owner.

## Candidate Moira Internal Modules

### Backend (Rust / Tauri side)

- `artifacts`
  - release fetch
  - checksum verification
  - install isolation
- `profiles`
  - JSONC profile storage
  - Core schema validation
  - active profile selection
- `supervisor`
  - wake/stop/force-kill
  - PID and run-state tracking
  - log receiver readiness gating
- `collector`
  - OTLP gRPC logs receiver
  - event normalization
- `store`
  - DuckDB connection, migrations, retention, query views
- `builder`
  - local source-folder validation
  - Core build invocation for development runs
  - build output discovery
- `query`
  - live subscription
  - indexed lookups
  - page-specific read models

### Frontend (Vue / Loom)

- `overview`
  - ingest health
  - current run state
  - recent errors
- `cortex`
  - tick list
  - tick detail
  - goal-forest change inspection
- `stem`
  - afferent/efferent signal stream
  - descriptor catalog
- `spine`
  - adapters
  - connected endpoints
  - dispatch outcomes
- `control`
  - installs
  - config profiles
  - wake/stop controls

## Candidate Local State Layout

This is a working shape only, not yet a durable contract.

- `installs/<core-version>/`
- `profiles/<profile-name>.jsonc`
- `runtime/current-run.json`
- `telemetry/moira.duckdb`
- `cache/releases/`

For the first landable Lachesis slice, the intended DuckDB shape is smaller:

- `raw_events`
- `runs`
- `ticks`

Selected tick detail should resolve from `raw_events` plus the `runs`/`ticks` indexes before adding narrower subsystem-specific projections.

## Candidate Data Flow

1. Loom asks Clotho for available artifacts and profiles.
2. Atropos wakes Core using a selected artifact + profile.
3. The selected profile points Core OTLP logs at Lachesis.
4. Lachesis ingests OTLP logs, normalizes Beluna-native fields, and writes raw + indexed rows into DuckDB.
5. Loom queries run/tick read models derived from DuckDB and subscribes to live updates for active runs.
6. Metrics and traces remain external handoff surfaces; Loom shows exporter state and links rather than collecting them locally.

## Candidate Structured Event Families Required From Core

These families are the minimum shape needed to support the requested observability views without parsing prompt-shaped blobs as the main contract.

### Cortex

- `cortex.tick`
  - run, tick, sense count, act count, continuation state
- `cortex.organ.request`
  - stage, tick, route, input shape summary, identifiers
- `cortex.organ.response`
  - stage, tick, response summary, tool summary, act summary
- `cortex.goal_forest.snapshot`
  - tick, snapshot handle, summary, full payload reference

### Stem

- `stem.signal.transition`
  - afferent/efferent, enqueue/defer/release/dispatch/result, descriptor identity, endpoint identity
- `stem.dispatch.transition`
  - act instance, tick, queue state, terminal outcome
- `stem.descriptor.catalog`
  - catalog version, adds, updates, drops

### Spine

- `spine.adapter.lifecycle`
  - adapter type, adapter id, state change
- `spine.endpoint.lifecycle`
  - endpoint id, channel id, connect/disconnect/register/drop
- `spine.dispatch.outcome`
  - act instance, endpoint, descriptor, dispatch binding, outcome, latency if available

## Decision Surfaces Still Open

- exact OTLP payload encoding policy for large bodies such as prompt input, output text, and goal-forest material
- how much of each tick should be denormalized into dedicated query tables versus derived from raw events
- DuckDB schema versioning and migration policy
- release manifest shape beyond GitHub asset discovery
- whether SHA256-only release verification is sufficient for this task or whether detached signatures should be added later
