# L2 Architecture Seams And Candidate Contracts

## Boundary Statement

- `core` remains authoritative for:
  - typed config schema
  - runtime behavior
  - endpoint routing and outcomes
  - observability emission
- `moira` becomes authoritative for:
  - local Core wake-input preparation and artifact management evolution
  - local Core supervision
  - OTLP log ingestion and storage
  - human-facing control and inspection UX
- `apple-universal` remains a body endpoint app, not a control-plane owner.

## Candidate Moira Internal Modules

### Backend (Rust / Tauri side)

- `app`
  - Tauri bootstrap
  - top-level module composition
  - command exposure
  - app-wide event wiring
- `clotho`
  - wake-input contract and preparation
  - known local build registration and resolution
  - release fetch
  - checksum verification
  - install isolation
  - local source-folder validation
  - Core build invocation for development runs
  - build output discovery
  - JSONC profile storage
  - Core schema validation
  - active profile selection
  - keep internal submodules functionally named, such as `artifacts` and `profiles`
- `lachesis`
  - OTLP gRPC logs receiver
  - event normalization
  - DuckDB connection, migrations, retention, query views
  - live subscription
  - indexed lookups
  - tick-lane read models and page-specific read models
- `atropos`
  - `wake`/stop/force-kill
  - PID and run-state tracking
  - log receiver readiness gating

### Frontend (Vue / Loom)

- `overview`
  - ingest health
  - current wake state
  - recent failures
- `tick-workspace`
  - tick list
  - per-tick gantt or lane chronology
  - structured drilldown
- `ai-gateway`
  - thread
  - turn
  - request lifecycle
  - committed-message inspection within turns
- `cortex`
  - organ inspection
  - goal-forest state and mutation history
- `stem`
  - stem tick
  - proprioception history
  - neural-signal descriptor history
  - afferent/efferent pathway flow
- `spine`
  - adapters
  - connected endpoints
  - dispatch binding and outcomes
- `control`
  - installs
  - config profiles
  - wake/stop controls

## Candidate Local State Layout

This is a working shape only, not yet a durable contract.

- `artifacts/known-local-builds/<build-id>.json`
- `installs/<core-version>/`
- `profiles/<profile-name>.jsonc`
- `runtime/current-run.json`
- `telemetry/moira.duckdb`
- `cache/releases/`

The next Lachesis increment should still keep `raw_events` authoritative, but it may add a narrow tick-browse projection if that clearly pays for the human-friendly lane view:

- `raw_events`
- `runs`
- `ticks`
- `tick_lanes` or an equivalent derived browse model if raw-only assembly proves too awkward

## Candidate Data Flow

1. Loom asks Clotho for available wake inputs and profiles.
2. Clotho resolves a selected known local build ref + optional profile ref into prepared wake input.
3. Atropos wakes Core using prepared wake input, without owning build-path details.
4. The selected profile points Core OTLP logs at Lachesis.
5. Lachesis ingests OTLP logs, normalizes Beluna-native fields, and writes raw + indexed rows into DuckDB.
6. Loom queries wake/tick/tick-lane read models derived from DuckDB and subscribes to live updates for active wakes.
7. Metrics and traces remain external handoff surfaces; Loom shows exporter state and links rather than collecting them locally.

## Candidate Structured Event Families Required From Core

This section was originally written as a target catalog before the current Core contract landed.
The current canonical family names now live in code and Moira projection already follows them.
Treat the list below as the working truth for this task buffer, and treat older task-buffer names such as `ai-gateway.turn`, `ai-gateway.thread`, `cortex.organ`, `stem.signal`, `stem.dispatch`, `stem.descriptor.catalog`, and `spine.dispatch` as superseded draft vocabulary.

Some fixture ids in `core/tests/fixtures/observability/*` still use older scenario labels such as `cortex.tick.nominal` or `stem.signal.transition.afferent`.
Those fixture ids are historical scenario names, not the current family catalog consumed by Moira.

### AI-Gateway

- `ai-gateway.request`
  - backend request lifecycle
  - retry and failure detail
  - full request/response payloads where available
  - terminal-state provider request payload retention when adapters cannot expose intermediate bodies cleanly
- `ai-gateway.chat.turn`
  - committed turn boundary
  - full turn messages array
  - usage, finish reason, backend metadata
- `ai-gateway.chat.thread`
  - authoritative thread snapshot
  - thread open or clone/reset visibility
  - post-turn persisted thread state

### Cortex

- `cortex.primary`
- `cortex.sense-helper`
- `cortex.goal-forest-helper`
- `cortex.acts-helper`
  - helper-scoped execution families that currently carry Cortex interval and AI-correlation detail
  - full input/output payloads
  - route or backend, request id, status, and error detail
- `cortex.goal-forest`
  - per-tick snapshot
  - patch or persist result payload
  - context-reset details when applicable

### Stem

- `stem.tick`
- `stem.afferent`
  - afferent sense movement, queueing, and release detail
  - full sense payload when relevant
- `stem.efferent`
  - efferent act movement and terminal result detail
  - full act payload when relevant
- `stem.proprioception`
  - patch/drop changes owned by the physical-state store
- `stem.ns-catalog`
  - snapshot/update/drop at catalog scope
- `stem.afferent.rule`
  - add/remove lifecycle for deferral rules

### Spine

- `spine.adapter`
  - lifecycle state
  - startup and fault visibility
- `spine.endpoint`
  - connect/drop lifecycle
  - channel/session visibility
- `spine.sense`
  - ingress sense visibility
  - endpoint and sense identity when present
- `spine.act`
  - act binding and terminal outcome visibility
  - endpoint, adapter, and act identity when present

## Decision Surfaces Still Open

- whether any adapter can cheaply expose provider-request payloads on every `ai-gateway.request` kind rather than only at terminal states
- whether any Stage 2 Loom view needs a secondary grouping inside one primary lane beyond the currently fixed lane matrix
