# L0 Intake Packet

## Perturbation

Upgrade the current `monitor` into `Moira`, a binary local control plane with:

- `Clotho`: Beluna Core download, hash verification, version isolation.
- `Lachesis`: OTLP logs receiving, storing, querying, and visualization.
- `Atropos`: wake and stop control for Beluna Core.
- `Loom`: the human-facing UI.
- Improved Core OTLP logs, especially for Cortex, Stem, and Spine observability.

The user explicitly prioritizes observability quality over generic raw-log or generic backend UI approaches.

## Input Type

- Primary: `Intent`
- Secondary: `Constraint`, `Artifact`

## Locked Decisions In This Round

- Quitting Moira also stops the supervised Core.
- Force-kill requires a second confirmation step.
- Config remains JSONC-only for now.
- Local development can point Clotho at a Core source folder and compile before launch.
- `tick` is the preferred product-facing name for the current `cycle_id` anchor.
- Goal-forest comparison is derived between two selected ticks and is not stored as a precomputed DB diff.
- Metrics and traces are not collected into Lachesis in this task; Loom only surfaces exporter status plus handoff links.
- Moira may eventually launch first-party endpoint apps, but that is not part of v1 supervision scope.
- `apple-universal` remains a pure body endpoint UX.
- Clotho trusts GitHub release assets using:
  - Core archive name pattern: `beluna-core-<rust-target-triple>.tar.gz`
  - current macOS-first asset: `beluna-core-aarch64-apple-darwin.tar.gz`
  - checksum file: `SHA256SUMS`

## Intended Change

Create a decomposition buffer that lets the team stabilize Moira scope, boundaries, sequence, and open decisions before implementation starts changing code and authoritative docs.

## Impact Hypothesis

- Primary hit:
  - introduce a new technical unit and container shape (`/moira`)
  - redefine the observability consumer/control-plane story
  - require structured Core OTLP log improvements
- Secondary hit:
  - retire or replace the current `/monitor`
  - update product TDD topology and cross-unit contracts
  - clarify how `apple-universal` coexists with a new control plane
- Confidence: medium-high
- Unknowns:
  - exact event model required for Cortex tick inspection
  - retention, migration, and compaction policy for embedded telemetry storage
  - whether Stage 5 should remain SHA256-only or later require detached signatures

## Temporary Assumptions

- Moira is macOS-first.
- Moira is a new `/moira` unit, not a rewrite-in-place of `/monitor`.
- Beluna Core release artifacts come from GitHub Releases, with prereleases shown first.
- Lachesis ingests OTLP logs first; metrics and traces are deferred from first-class storage/UI.
- Full raw OTLP log payloads are retained locally during early development.
- Moira only supervises Beluna Core in v1.

## Negotiation Triggers

- Any attempt to shift runtime authority from `core` into Moira.
- Any pressure to couple observability deeply to generic third-party UI semantics instead of Beluna-native views.
- Any storage or event-model shortcut that would materially reduce Cortex/Stem/Spine inspectability.
- Any implementation shortcut that degrades readability or maintainability in the long-lived Moira/Core boundary.

## Acceptance Criteria

- The task is decomposed into bounded workstreams with explicit dependencies.
- The packet states what is assumed, what is open, and what must be promoted later.
- Observability-first sequencing is made explicit instead of being buried under packaging/bootstrap work.
- The packet gives enough structure that future implementation tasks can be split without redoing L0 alignment.
