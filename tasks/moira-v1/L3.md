# L3 Implementation Stages For This Task

All stages below are implementation stages of this single Moira task. They are not separate product phases.

## Stage 0. Task Buffer And Scaffold

### Objective

Finish the docs-first workflow and lock the implementation packet before code changes begin.

### Likely Paths

- `tasks/moira-v1/*`

### Done When

- `L0`, `L1`, `L2`, `L3`, and `LACHESIS` are coherent enough to guide implementation
- assumptions, defaults, and unresolved questions are explicit
- no authoritative docs are half-updated

## Stage 1. Lachesis Minimal Ingest

### Objective

Start code implementation with the minimum Moira scaffold required to receive OTLP logs from Core, store them, and support wake- and tick-scoped inspection.

### Scope

- `moira/*` scaffold
- OTLP gRPC logs receiver
- DuckDB `raw_events` table
- DuckDB `runs` projection
- DuckDB `ticks` projection
- minimal query API
- Loom wake list
- Loom tick list
- Loom selected tick workspace
- raw-event inspector for the selected tick

### Done When

- a running Core can stream logs into Moira
- operators can inspect one wake and one selected tick from Moira without external collector/UI tooling
- selected tick detail is reconstructed from raw events plus `runs`/`ticks`, not from a large precommitted projection lattice

### Stage Gate

- confirm which missing Core fields actually block humane tick inspection before introducing more tables
- do not ask release-artifact questions here

## Stage 2. Core Event Contract Realignment

### Objective

Replace the too-thin Stage 1 event contract with the richer logical families Lachesis actually needs.
This stage now follows the canonical names already landed in Core code, not the older draft names used in the first task packet.

### Scope

- AI-gateway events:
  - `ai-gateway.request`
  - `ai-gateway.chat.turn`
  - `ai-gateway.chat.thread`
- Stem events:
  - `stem.tick`
  - `stem.afferent`
  - `stem.efferent`
  - `stem.proprioception`
  - `stem.ns-catalog`
  - `stem.afferent.rule`
- Cortex events:
  - `cortex.primary`
  - `cortex.sense-helper`
  - `cortex.goal-forest-helper`
  - `cortex.acts-helper`
  - `cortex.goal-forest`
- Spine events:
  - `spine.adapter`
  - `spine.endpoint`
  - `spine.sense`
  - `spine.act`
- full request, response, sense, act, thinking, and topology payload preservation in raw events
- explicit `tick`, span, and lane correlation so the selected tick can be rendered as chronology rather than a flat list
- owner-centric semantics in `kind`, `status`, and `transition_kind` fields instead of a large per-verb family catalog

### Done When

- Stem and Spine are not empty because of fuzzy keyword grouping
- organ inspection includes full input/output evidence
- AI-gateway browsing no longer depends on Cortex summaries pretending to be conversation history
- request retries or failures are visible at the AI-gateway boundary without inventing synthetic conversation events
- legacy draft names or non-entity lane hints are no longer treated as authoritative model shape
- the canonical docs and Core emit points agree on the new owner-centric family catalog

### Stage Gate

- keep derived storage minimal while the new event contract settles
- add indexes only where the browsing model proves the need

## Stage 3. Loom Human-Friendly Browsing

### Objective

Replace the raw-browser-first UI with a humane tick workspace.

### Scope

- lane-based per-tick chronology or Gantt-like rendering
- conversation browser backed by authoritative thread snapshots, committed turns, and linked request lifecycle
- selected event detail cards
- Cortex / Stem / Spine drilldown that reflects the real event families

### Done When

- a human can understand one selected tick without reading raw JSON first
- the same tick workspace can pivot between chronology, conversation, subsystem detail, and raw fallback

### Stage Gate

- if browsing still requires reparsing raw payload blobs in the view layer, add only the minimal supporting indexes

## Stage 4. Atropos Minimal Supervision

### Objective

Wake and stop a selected local Core build under Moira control.

### Scope

- launch command construction
- receiver-ready gating
- PID/run-state tracking
- graceful stop and explicit force-kill
- app-exit behavior that stops supervised Core

### Done When

- Moira can start a local Core build, observe it, and stop it deterministically

### Stage Gate

- no new release-manifest questions until supervision works with a known local build

## Stage 5. Clotho Wake Preparation And Artifact Evolution

### Objective

First establish a stable Clotho-owned wake-input boundary, then evolve it toward broader artifact management.

### Scope

First slice:

- wake-input contract
- known local build ref registration and resolution
- profile resolution into prepared wake input
- keep Atropos consuming prepared wake input instead of raw build paths

Later slice:

- local source-folder compile workflow
- GitHub release discovery
- checksum verification
- version-isolated install directories
- selected artifact binding in Atropos wake path

### Done When

- first supervised wake can use a Clotho-prepared known local build ref without Atropos owning build-path details
- Moira can compile from a local Core source folder for development
- Moira can install, verify, select, and run a specific Core version

### Stage Gate

- do not let release-distribution work block the first Clotho-prepared local wake path
- release asset naming, checksum file naming, and trust contract are decided immediately before the later artifact-management slice

## Stage 6. Expansion And Hardening

### Objective

Expand the observability surfaces that clearly justify themselves and then harden the result.

### Scope

- dedicated Goal-forest compare
- dedicated Stem view if needed beyond the tick workspace
- dedicated Spine topology/dispatch view if needed beyond the tick workspace
- JSONC profile management
- Core schema validation
- authoritative doc promotion and deployment notes

### Done When

- the requested explainability primitives are visible without leaving Moira
- the same workflow can be repeated across multiple Core versions and profiles
- stable boundaries are promoted into `20-product-tdd`, `30-unit-tdd`, and `40-deployment`

## Stage Rule

Each stage should be landable on its own. Avoid stages that simultaneously introduce:

- a new durable boundary
- a new runtime surface
- a new storage contract
- and a new UI abstraction

That combination would hide risk instead of isolating it.

During MVP, do not make broad automated test additions a stage gate by default. Prefer live end-to-end checks until the new event contract and operator surfaces stop churning.
